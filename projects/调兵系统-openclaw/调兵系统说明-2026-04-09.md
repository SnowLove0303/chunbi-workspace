# 调兵系统说明

日期：2026-04-09

## 一、系统定位

调兵系统的核心目标是让 `Codex` 作为总司令，负责：

- 判断任务类型
- 选择合适兵团
- 决定是否扩兵
- 分派任务
- 验收结果
- 汇总最终输出

`Codex` 负责指挥和验收，不负责重写三个兵原本已经具备的 Agent 能力。

## 二、总框架（四层）

### 1. 总控层

`Codex` 作为独立总司令，原则是"最小必要调度"代替"过度包装"。

### 2. 兵团层

三个主力兵团：

- **CoPaw** — 协作型，多 agent 并行，宽松自然语言任务
- **OpenClaw** — 结构化 Agent 平台，原生多 agent / sessions / spawn，适合重型编排
- **OpenCode** — 执行型，编码/成稿/精确输出，多角色多端口

### 3. 扩编层

每个兵团都可以同时调多个 agent，并支持两种扩编方式叠加：

- `Codex` 直接点名多个 agent 并行执行
- 兵团自己再开 agent 执行子任务

### 4. 工作区层

每个 agent 拥有独立 workspace，保证多 agent 并行时上下文不互相覆盖。

## 三、当前已实现的主要能力

### 1. 基础调兵

单兵 / 双兵 / 三兵并行 / 结果整合，通过 `assistant-orchestrator` 或 `assistant-hub.ps1` 入口。

### 2. 直接点将

直接指定具体 agent / 具体角色 / 直接并行点名，不再把所有请求交给兵团内部自己拆。

### 3. 多兵并行

- `OpenCode` 多角色并行（`exact` / `writer` / `reviewer`）实测通过
- `CoPaw` 多 agent 并行可用
- `OpenClaw` 指定 agent 点将可用

### 4. 重型多 Agent 工作流

`ClawTeam` 已接入完整能力：

- 模板工作流
- team 启动 / 概览 / 任务查看
- inbox / message 查看
- board 看板
- `openclaw-heavy` 重型编队模式

## 四、推荐调度原则

1. **默认先少量用兵** — 默认先用 1 个 agent，判断复杂再扩兵
2. **一旦一个不够就扩兵** — 需要对比 / 并行提速 / 角色分工 / 交叉验收时
3. **把结果整理交给专门角色** — 前端 agent 正常产出，后端专门整理 agent 收口
4. **优先用原生能力** — 不过度包装，不重写兵团原生内部机制

## 五、主要入口

| 类型 | 路径 |
|------|------|
| 本地技能入口 | `F:\codex\.codex\skills\assistant-orchestrator\SKILL.md` |
| 本地命令入口 | `F:\codex\codex\tools\assistant-hub.ps1` |
| CMD 包装 | `F:\codex\codex\tools\assistant-hub.cmd` |
| ClawTeam 入口 | `F:\codex\codex\tools\clawteam-openclaw.ps1` |

## 六、已确认的能力边界

### 可用

- CoPaw 单任务 / 多 agent 并行
- OpenCode 多角色并行
- OpenClaw 直接点 agent
- ClawTeam 模板和看板
- `openclaw-heavy` 编队模式

### 仍需优化

- CoPaw 历史不健康 agent 的整理
- OpenClaw 某些 Windows 运行态稳定性
- 飞书桥中文编码问题
- 不同兵团结果做更统一的汇总格式

## 七、当前最准确的一句话总结

`Codex` 总控 + 3 个兵团 + 可扩编 agent + 独立 workspace + 重型多 agent 工作流，当前最重要的优化方向是让结果汇总、编码稳定性和兵团健康度更统一。
